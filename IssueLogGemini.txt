   I've investigated further and found the final piece of the puzzle. The issue isn't just about memory—it's a thread-blocking problem directly related to how the application
  updates the UI after a search result is clicked.


  What caused the issue:
  When you select a search result from the QuickSearch dropdown, the application calls ExplorerTopComponent.setExplorerRoot(node) on the Java UI Thread (EDT) to update the
  explorer view.


  Inside that method, the code tries to automatically expand the tree to show the parents and children of the selected node by calling getNodes(true). The true parameter
  explicitly tells NetBeans to block the thread and wait until all the child nodes are fully calculated from the Neo4j database. Since this was running on the UI thread, the
  entire macOS interface froze while it queried the database and built the tree hierarchy for the selected SNOMED CT concept.


  How it was fixed:
  I rewrote the setExplorerRoot method in ExplorerTopComponent.java to perform the heavy lifting on a background thread:


   1. The method now pushes the blocking getNodes(true) calculations to NetBeans' background RequestProcessor (RP.post(...)).
   2. The UI thread is immediately freed up, allowing the QuickSearch popup to dismiss smoothly without freezing the app.
   3. Once the background thread finishes calculating the tree hierarchy from Neo4j, it safely schedules just the final visual expansion (btv.expandNode) back onto the UI
      thread.

I have investigated and resolved the issue where inferred relationships and their hierarchy were missing from the Concept Details Window.


  What caused the issue:
  The application was silently failing when trying to read the inferred relationships from the local Neo4j database.
  In the QuerySCTimpl.java database querying logic, the application checked if a relationship was inferred by attempting to cast the database property directly to a Java
  integer like this: (int) r.getProperty("stated") == 0.


  However, depending on how the SNOMED CT RF2 release was imported, the Neo4j database engine often optimizes and returns numeric properties as Long objects instead of Integer.
  When this happens, Java throws a silent ClassCastException in the background, which aborts the processing of the tree nodes (hierarchy) and the relationship models for the
  ConceptDetailsWindow.


  Because stated relationships were already using a safer helper method (matchesIntProperty) to check properties, they were unaffected by this bug.


  How it was fixed:
   1. I wrote a Python script (fix_class_cast.py) to systematically update QuerySCT_Impl/src/org/ihtsdo/mysnow/querysct_impl/QuerySCTimpl.java.
   2. I replaced all fragile (int) r.getProperty("stated") == 0 casts with the robust matchesIntProperty(r, "stated", 0) method, which properly handles Long/Integer variations
      as well as gracefully handles missing properties.
   3. I recompiled the QuerySCT_Impl module and packaged the full macOS application again using ./tools/build-mac-fixed.sh.
